# consensus-specs — directory structure (deep-dive)

The actual spec test files sit nine path components deep, mixed with
five different infrastructure layers spread across as many
directories. The repo's stated purpose is "write specs and write
tests against them"; the layout makes both hard to find. This guide
lays out what's there, contrasts it with cleaner layouts in
comparable projects, and proposes a target structure organised
around four conceptually distinct buckets.

## What's there today

### Top-level

The repo root has eight content directories plus the usual housekeeping:

```
consensus-specs/
├── configs/         ← yaml runtime config
├── fork_choice/     ← markdown spec text
├── presets/         ← yaml compile-time config
├── pysetup/         ← markdown→Python builder
├── scripts/         ← ad-hoc check tools
├── specs/           ← markdown spec text (per-fork)
├── ssz/             ← markdown spec text
├── sync/            ← markdown spec text
├── tests/           ← everything test-related
├── setup.py, pyproject.toml, Makefile, README.md, AGENTS.md, …
```

Six of those eight directories (`configs/`, `fork_choice/`, `presets/`,
`specs/`, `ssz/`, `sync/`) are spec content. They sit at the root as
peers of `pysetup/` and `tests/`. A first-time reader looking at the
root cannot tell from directory names alone which dirs are content
and which are tooling.

### Inside `tests/`

Five overlapping concerns share the `tests/` tree:

```
tests/
├── checklists/         ← single markdown file (heze.md)
├── core/pyspec/eth_consensus_specs/   ← THE TEST CODE — and the
│   │                                    Python package it imports
│   ├── test/<fork>/<category>/test_*.py   ← real tests, deep
│   ├── test/helpers/   ← 59 helper modules, ~10k lines
│   ├── utils/          ← bls.py, kzg.py, ssz/, hash_function.py
│   ├── config/, debug/ ← spec runtime utilities
│   └── __init__.py, py.typed, VERSION.txt
├── formats/            ← markdown describing YAML test-vector formats
├── generators/         ← fork-choice constraint solver + runners +
│   │                     gen_base/ helpers
│   ├── compliance_runners/fork_choice/
│   └── gen_base/
└── infra/              ← pytest plugins, manifest, dumper, randomized
    ├── helpers/          blocks; PLUS its own tests of itself
    ├── pytest_plugins/
    ├── *.py              (yield_generator.py, manifest.py,
    │                      dumper.py, block_randomized.py,
    │                      template_test.py)
    └── test_*.py         ← tests of the test infrastructure
```

The five concerns inside `tests/`:

1. `checklists/` — manual fork-coverage tracking (one file, currently
   only Heze).
2. `core/pyspec/eth_consensus_specs/` — the runtime spec package
   (gitignored, generated by `pysetup`) plus the actual test files
   plus the helper modules plus utility modules. Four logically
   different things in one tree.
3. `formats/` — markdown defining the YAML/SSZ test-vector format
   for cross-client consumption.
4. `generators/` — constraint solvers and runners that synthesise
   test vectors (specifically for fork-choice).
5. `infra/` — pytest plugins, the yield-collection machinery, the
   YAML/SSZ dumper, the fixture manifest, randomized-block generation,
   plus tests of all of these.

### How deep are the actual tests?

A representative path:

```
consensus-specs/tests/core/pyspec/eth_consensus_specs/test/phase0/sanity/test_blocks.py
   1            2     3    4      5                    6    7      8      9
```

Nine path components from the repo root to a real test file. 215 of
the 228 test files in the repo live at this depth.

The breakdown:

- `tests/core/pyspec/eth_consensus_specs/` is *both* a directory and an
  installable Python package; `eth_consensus_specs` is the package
  name, mapped to this path by `setup.py:19–25`. The `test/` (singular)
  subdirectory inside the package is the spec test layer; the package
  also contains `helpers/` (~9 950 lines of test infrastructure mixed
  in), `utils/`, `config/`, and `debug/`.
- 59 helper modules sit as direct siblings of the per-fork test trees
  inside the same package.
- The `tests/` (plural) directory therefore mixes the package being
  tested with the tests of that package — and the test infrastructure
  with the tests of the test infrastructure (`tests/infra/test_*.py`).

## What the comparables do

Both comparable repos use a clean three-directory model.

### `execution-specs/`

```
execution-specs/
├── src/ethereum/forks/<fork>/...      ← spec authored as Python (WET)
├── packages/testing/src/execution_testing/
│       └── (cli, plugins, test_fixtures, forks, ...)   ← framework
├── tests/<fork>/<category>/test_*.py  ← tests, 4 levels deep
└── pyproject.toml, Justfile, README.md, …
```

A real test file: `tests/osaka/eip7825_transaction_gas_limit_cap/test_*.py`
— four path components. The spec is at `src/ethereum/forks/osaka/`;
the framework is at `packages/testing/src/execution_testing/`. Each of
the three concerns has one home.

### `leanSpec/`

```
leanSpec/
├── src/lean_spec/                      ← spec authored as Python (Pydantic)
│   ├── types/, subspecs/, forks/, …
├── packages/testing/src/{framework,consensus_testing}/
│       └── pytest_plugins/, test_fixtures/, forks/, …  ← framework
├── tests/<area>/<subarea>/test_*.py    ← tests, 4 levels deep
└── pyproject.toml, tox.ini, README.md, …
```

Same three-directory model. A test file lives at, e.g.,
`tests/lean_spec/forks/lstar/forkchoice/test_*.py` — four levels with
optional sub-grouping.

### What both comparables share

- **Three top-level content directories**: `src/` (spec), `packages/`
  (framework), `tests/` (tests). The "where do I find X?" question
  has at most three answers.
- **Test files at 3–4 levels of depth.** Not nine.
- **No mixing of test code with the package being tested.** The
  package the tests import lives in a different directory than the
  tests themselves; nobody has to think about which import path
  applies.
- **One canonical place for test helpers.** Inside the `packages/
  testing/` workspace member. Not co-mingled with the test files.

## Evaluating the three-bucket framing

The natural framing — and the one suggested when this question was
raised — is that the repo has three concerns: **specs**, **infra-
structure** (pytest plugins + the markdown→Python builder), and
**tests**. That maps cleanly to most of what's there, but three
places need refinement before it becomes a target structure:

1. **The generated executable spec is a fourth bucket.** The markdown
   in `specs/` is the source of truth, but the runtime that tests
   actually import is the *generated* `eth_consensus_specs` package
   (built into `tests/core/pyspec/...`). It's neither markdown spec
   nor build infrastructure — it's the spec-as-Python, currently
   produced as a build artefact in a gitignored location. It needs a
   clean home rather than living buried in the test tree.

2. **The test-vector formats (`tests/formats/`) need a place.**
   Markdown defining YAML/SSZ structure for cross-client test vectors.
   Not test code, not infrastructure code — they're a *spec of test
   vectors*, and the natural home is alongside the rest of the spec
   markdown.

3. **The fork-choice generator is a different shape of infrastructure.**
   A constraint solver, instantiators, and a runner — distinct from
   the pytest-plugin layer. In the comparables this kind of test-
   vector synthesis lives inside `packages/testing/...` under a
   sub-namespace. Folding it under one infrastructure umbrella works
   as long as it gets a clear sub-directory.

## Recommended target structure

```
consensus-specs/
├── specs/                            ← spec source (markdown)
│   ├── phase0/, altair/, bellatrix/, …
│   ├── presets/                      ← (was /presets at root)
│   ├── configs/                      ← (was /configs at root)
│   ├── ssz/, fork_choice/, sync/     ← (were at root)
│   └── formats/                      ← (was tests/formats/)
│
├── src/eth_consensus_specs/          ← EXECUTABLE SPEC (generated)
│   ├── phase0/, altair/, …
│   ├── utils/, config/, debug/       ← (were tests/core/pyspec/.../)
│   └── (no test/ subdir — tests move out)
│
├── framework/                        ← INFRASTRUCTURE (single home)
│   ├── builder/                      ← was pysetup/
│   ├── plugins/                      ← was tests/infra/pytest_plugins/
│   ├── runtime/                      ← yield_generator, dumper, manifest
│   ├── helpers/                      ← was tests/core/pyspec/.../helpers/
│   ├── generators/                   ← was tests/generators/
│   └── tests/                        ← tests OF the framework
│                                       (was tests/infra/test_*.py)
│
├── tests/                            ← TESTS THEMSELVES, flat
│   ├── phase0/sanity/test_blocks.py  ← 4 levels deep, not 9
│   ├── altair/, bellatrix/, …
│   └── checklists/                   ← (was tests/checklists/)
│
├── docs/, scripts/, .github/, …
└── pyproject.toml, README.md, AGENTS.md, Makefile, …
```

The four conceptual buckets:

| Bucket | Today | Target |
|---|---|---|
| Spec source (markdown) | `specs/`, `presets/`, `configs/`, `ssz/`, `fork_choice/`, `sync/`, `tests/formats/` | `specs/` (consolidated) |
| Executable spec (generated Python) | `tests/core/pyspec/eth_consensus_specs/{phase0,altair,…}/{minimal,mainnet}.py` | `src/eth_consensus_specs/` |
| Framework (pytest plugins, helpers, builder, generators) | `pysetup/`, `tests/infra/`, `tests/core/pyspec/.../{helpers,utils,config,debug}/`, `tests/generators/` | `framework/` |
| Tests | `tests/core/pyspec/eth_consensus_specs/test/<fork>/<category>/test_*.py` | `tests/<fork>/<category>/test_*.py` |

### Path depth comparison

| Item | Today | Target |
|---|---|---|
| Spec test file | 9 levels | 4 levels |
| Spec source (markdown) | 3 levels | 3 levels (unchanged) |
| Helper module | 8 levels (mixed with tests) | 3 levels (under `framework/helpers/`) |
| Pytest plugin | 4 levels | 3 levels (`framework/plugins/`) |
| Build tooling | 2 levels | 3 levels (`framework/builder/`) |
| Generated runtime spec | 6 levels (gitignored) | 3 levels (under `src/`) |

## Why this matters concretely

- **A new contributor's first question is "where do I add a test?"**
  Today the answer requires understanding nine path components, the
  decorator stack, and the file-naming convention that the test-vector
  generator uses to derive the manifest. With the target it is "make
  a file at `tests/<fork>/<category>/test_X.py`" — same as
  `execution-specs` and `leanSpec`.
- **The infrastructure stops bleeding into the tests.** Today, finding
  the helper for "build an attestation" or "transition to a state"
  requires knowing the helper lives in a sibling directory of the
  tests *inside the same Python package*. With one `framework/helpers/`
  the answer is one directory, one grep.
- **Spec readability.** The markdown spec lives at `specs/` already,
  which is fine; consolidating `presets/`, `configs/`, `ssz/`,
  `fork_choice/`, `sync/`, `formats/` under it makes the "where is
  the spec?" answer obvious.
- **The package-being-tested stops being inside the tests directory.**
  This is the structural fix to the self-referential layout problem
  ([self-referential-package-layout.md](self-referential-package-layout.md)):
  with the package at `src/eth_consensus_specs/` and the tests at
  `tests/`, the dual-import pathology cannot occur because the test
  directory and the package source directory are different
  directories.
- **Tooling stops needing per-project workarounds.** Pyright,
  IDE jump-to-definition, coverage, and ruff all behave better with a
  conventional `src/` layout. The `pythonpath = ["."]` workaround in
  `pyproject.toml:60–61` becomes unnecessary; the gitignored
  per-fork enumeration in `.gitignore:18–27` collapses; the
  build-time `make _pyspec` precondition for `pip install .` goes
  away.

## What it dissolves

This refactor isn't only a layout cleanup; it removes the cause of
several already-listed findings as side-effects:

- The §1 "Preset swapping with dual-module imports is fragile"
  finding goes away (one canonical import path).
- The §2 "Tests of tests coexist with production infrastructure in
  tests/infra/" finding goes away (tests of the framework live at
  `framework/tests/`, separate from the test tree).
- The §2 "Overlapping helpers in tests/infra/helpers and
  tests/core/pyspec/.../helpers" finding goes away (one
  `framework/helpers/`).
- The §3 god-helper-layer findings get a different shape: the helpers
  at `framework/helpers/` can be properly typed and split because they
  are no longer mixed with test files in the same directory tree.
- The §4 "Per-fork test layout & duplication" findings change shape:
  with `tests/<fork>/<category>/`, the matrix becomes legibly flat
  rather than buried under five wrapping directories.
- The §5 "Fork-choice generator is a divergent subsystem" finding gets
  a home: `framework/generators/fork_choice/` makes its place in the
  framework explicit.
- The §6 "`setup.py` defers all spec generation to `make _pyspec`"
  finding gets a path forward: with `src/eth_consensus_specs/` as the
  build target, a PEP 517 build backend can run the generation as
  part of `pip install`.
- The §9 "`.gitignore` exhibits Shotgun Surgery for generated phases"
  finding goes away (or becomes a single `src/eth_consensus_specs/<fork>/`
  glob).

## Honest caveats

- **It is a major refactor.** Touching `setup.py`, `pyproject.toml`,
  every test file's imports, the `pysetup` retargeting,
  `Makefile` / CI / `.gitignore`, and downstream consumers of test
  vectors. Complex, but the transition plan above lets each step
  land independently rather than as one big-bang change. This is
  the kind of work that justifies a v2 release.
- **It is intertwined with the self-referential-package-layout fix.**
  You cannot move the package out of `tests/` and *also* leave tests
  buried 9 levels deep — the two refactors share the same lift.
  Whichever is tackled first sets the shape for the other.
- **Spec authoring choice is upstream of layout.** The deepest source
  of pain is "spec lives in markdown, gets generated to Python at
  build time". `leanSpec` and `execution-specs` avoid this entirely
  by writing the spec as Python directly, which is why their layouts
  are simpler. If consensus-specs ever moves in that direction, the
  layout question becomes easier (no `src/eth_consensus_specs/`
  generated-output bucket — just `src/` as authored). That's a
  separate, much bigger decision; the layout fix described here does
  not require it.
- **Some path conventions are pinned externally.** Test-vector
  consumer tools in other clients read the YAML output paths;
  restructuring needs to either preserve those or be coordinated
  with consumers. A cosmetic-only move at the source level can keep
  the artefact paths unchanged.
- **`tests/` is one inversion away from looking like every other
  Python project.** The thing that's strange is not that there's a
  `tests/` directory — there is one — it's that the test files live
  inside a package nested inside `tests/`. Inverting that inversion
  is what makes everything else easier.

## Named anti-patterns

- **Misplaced Class** (Fowler, *Refactoring*, 1999) at the package
  level — modules end up in directories that don't reflect their
  responsibility (helpers under `tests/infra/` and again under
  `tests/core/pyspec/.../`; spec markdown spread across six root
  dirs).
- **Inappropriate Intimacy** (Fowler, p. 85) at the directory level
  — the `tests/` directory knows that `eth_consensus_specs` is a
  Python package, knows where its source lives, and exposes that
  layout to every consumer.
- **Shotgun Surgery** (Fowler, p. 79) for any cross-cutting change —
  a fork rename touches `pysetup/`, `Makefile`, CI, `.gitignore`,
  multiple test directories, plus the package init.
- **Speculative Generality** (Fowler, p. 109) at the `tests/`
  organisation — the `tests/core/pyspec/eth_consensus_specs/test/`
  nesting was set up to accommodate other "core" things that never
  arrived; it's structure for a generality that didn't materialise.
- **Hierarchical Mismatch** (Lakos, *Large-Scale C++ Software
  Design*, 1996, ch. 4) — the physical hierarchy (directories) does
  not reflect the logical hierarchy (which things depend on which).
  In a healthy layout, dependencies flow downward through the
  hierarchy; here the test directory contains the package being
  tested.

## What fixing it would entail

A practical staging:

1. **Move the `eth_consensus_specs` package out of `tests/`.** Decide
   whether the generated runtime lives at `src/eth_consensus_specs/`
   (PEP 517 build backend regenerates it) or at a non-`src/` location
   if the team prefers to keep the build-as-Make pattern.
2. **Flatten the test tree.** `tests/<fork>/<category>/test_*.py`,
   no `core/pyspec/eth_consensus_specs/test/` wrapping. Update every
   test file's imports to reference the now-canonical
   `eth_consensus_specs` package by its single name.
3. **Drop `pythonpath = ["."]`** from `pyproject.toml`. Tests reach
   the package via the editable install only.
4. **Consolidate the framework.** Move `pysetup/` → `framework/builder/`,
   `tests/infra/` → `framework/{plugins,runtime}/`,
   `tests/core/pyspec/.../helpers/` → `framework/helpers/`,
   `tests/generators/` → `framework/generators/`. Move the
   tests-of-infrastructure (`tests/infra/test_*.py`) to
   `framework/tests/`.
5. **Consolidate the spec source.** Move `presets/`, `configs/`,
   `ssz/`, `fork_choice/`, `sync/`, `tests/formats/` under `specs/`.
   Update `pysetup` paths accordingly.
6. **Delete the now-redundant workarounds.** The conftest dual-mutation
   (`conftest.py:113–120`), the per-fork `.gitignore` lines, the
   `setup.py` `package_dir` mapping, and the install-requires-Make
   precondition can all go away or simplify dramatically.

That's a non-trivial sequence — but each step is mechanically
straightforward, and several steps fix multiple findings at once.
