# consensus-specs — SSZ generic vectors (deep-dive)

The test-vector generation system at
`tests/core/pyspec/eth_consensus_specs/test/phase0/ssz_generic/` has
two notable properties: (1) the test framework uses the SSZ library
*as its own oracle for validity* — a tautological pattern that
publishes the library's behaviour as an inter-client conformance
spec; (2) the ~1 313 lines of generation infrastructure form a
parallel ad-hoc system to
[compliance-runners.md](compliance-runners.md), but with a different
shape (random / systematic generation rather than constraint-solver
synthesis).

Adjacent guides:
[compliance-runners.md](compliance-runners.md) (sibling generation-
vs-validation system),
[decorator-stack.md](decorator-stack.md) (the seven-decorator chain
each generated test wears),
[markdown-as-source-of-truth.md](markdown-as-source-of-truth.md)
(`tests/formats/ssz_generic/README.md` is the third source of truth),
[helper-layer.md](helper-layer.md) (`template_test`'s frame-inspection
registration is the helper layer's largest user).

## The shape of the problem

`ssz_generic/` produces a published reference suite of SSZ test
vectors: `valid` cases (a value plus its serialised bytes plus its
hash-tree-root) and `invalid` cases (a byte string the library refuses
to deserialise). Other Ethereum clients consume these vectors as part
of their SSZ conformance testing. If the Python implementation in
`eth_consensus_specs.utils.ssz` has a bug, the published vectors
encode the bug as the spec.

Two patterns intertwine to produce the system:

### Pattern 1: The library is its own oracle

For valid cases, the test asserts only that
`deserialize(serialize(value)) == value` — a **round-trip property**.
This catches the case where `serialize` and `deserialize` disagree, but
does not catch the case where they agree on the *wrong* bytes (e.g.
both implementations skip a field, both pad the wrong way, both encode
endianness consistently but incorrectly).

For invalid cases, the test asserts that the library *raises* on the
generated bytes. The bytes are then published as "this is what an
invalid X looks like." If the library is **too strict** (rejects
valid bytes), the published vector tells other clients to reject
bytes those clients correctly accept, and the published vector is
*wrong*. If the library is **too permissive** (accepts truly invalid
bytes), the test fails — but only for the bytes the test author
happened to generate. There is no enumeration of all the ways
something could be invalid; the test just samples from a
hand-constructed set of "looks invalid" patterns.

The fundamental shape: **the test's oracle for validity is the
library's behaviour**. The vectors get their authority from
cross-client differential testing *after* publication, not from the
test framework itself. If a vector's "ground truth" is wrong because
the Python library was wrong, every client that disagrees is the one
that gets blamed.

### Pattern 2: Ad-hoc generation infrastructure

Around that core, ~1 313 lines of bespoke machinery: a 78-line driver
that registers tests via frame inspection, a 57-line case framework
defining `valid_test_case` / `invalid_test_case` factories, ten
type-specific cases files (each ~50–250 lines) that re-implement
similar iteration patterns (types × random modes × lengths × edge
cases). Each generated test wears a stack of seven decorators:
`@template_test` on the outside, then `@manifest`, `@only_generator`,
`@with_phases([PHASE0])`, `@spec_test`, `@single_phase`, plus the
inner generator function — most of which exist only because the
project's pyspec test machinery requires them
([decorator-stack.md](decorator-stack.md)).

## Proof, by line

### The tautological oracle (`ssz_test_case.py`)

The whole framework is 57 lines. The two factories are the headline:

`ssz_test_case.py:18–28` — `valid_test_case`:

```python
def valid_test_case(value_fn: Callable[[], View], rng: Random = None):
    seed = capture_seed(rng)

    def case_fn():
        if seed is not None:
            value = safe_lambda(value_fn)(rng=Random(seed))
        else:
            value = safe_lambda(value_fn)()
        serialized = serialize(value)
        assert deserialize(value.__class__, serialized) == value
        yield "value", "data", encode(value)
        yield "serialized", "ssz", serialized
        yield "root", "meta", "0x" + hash_tree_root(value).hex()

    return case_fn
```

The only correctness assertion is round-trip equality (line 25).
Everything yielded after that — the typed `value`, the `serialized`
bytes, the `hash_tree_root` — is taken from the library and emitted as
the canonical answer.

`ssz_test_case.py:31–48` — `invalid_test_case`:

```python
def invalid_test_case(typ: type[View], bytez_fn: Callable[[], bytes], rng: Random = None):
    seed = capture_seed(rng)

    def case_fn():
        if seed is not None:
            serialized = safe_lambda(bytez_fn)(rng=Random(seed))
        else:
            serialized = safe_lambda(bytez_fn)()
        try:
            _ = deserialize(typ, serialized)
        except Exception:
            yield "serialized", "ssz", serialized
            return
        code = bytez_fn.__code__
        raise ValueError(
            f"Invalid {typ.type_repr()} data should not deserialize: {serialized.hex()} in {code.co_filename}:{code.co_firstlineno}"
        )

    return case_fn
```

The contract: "the bytes I label invalid will be rejected by the
library." If the library *also* thinks they're invalid, the vector is
published. If the library disagrees, the test fails. The test does
not check whether the bytes are *truly* invalid by the SSZ
specification — only whether the library agrees with the author's
label.

The `try/except Exception` is also extremely broad. Any exception
type — `IndexError`, `OverflowError`, a stray `KeyError` from a
library bug, a `MemoryError` if the bytes claim a huge length —
counts as "the library rejected this." Granular exception types
(`SSZDeserialisationError`, `SSZLengthError`, `SSZValueError`)
would let the test author specify *why* the bytes are invalid.

**Note: the breadth is downstream of `remerkleable`'s exception
model.** The SSZ library underlying `consensus-specs`
(`eth-remerkleable == 0.1.30`, distributed as `remerkleable`)
raises `Exception` subclasses without a structured taxonomy of
failure modes — there is no public `SSZDeserialisationError` /
`SSZLengthError` / `SSZValueError` hierarchy that test code can
match against. The `except Exception:` here is the only viable
catch given the library's surface; tightening it requires either
upstream changes to `remerkleable` or a wrapper layer that
classifies the underlying errors. By contrast, `leanSpec`'s
in-repo SSZ raises typed `SSZTypeError`, `SSZValueError`,
`SSZSerializationError` from `lean_spec.types.exceptions`, with
construction-time validation that catches malformed inputs at the
earliest possible point. Calling out the dependency here so the
shape's origin is explicit; the refactor sketched below is most
naturally done together with a wrapper that classifies remerkleable's
errors.

### The decorator chain on each generated test (`test_ssz_generic.py:30–50`)

```python
@template_test
def _template_ssz_generic_test(handler_name, suite_name, case_name, case_fn):
    _manifest = Manifest(
        preset_name="general",
        runner_name="ssz_generic",
        handler_name=handler_name,
        suite_name=suite_name,
        case_name=case_name,
    )

    @manifest(_manifest)
    @only_generator("ssz_generic test for reference test generation")
    @with_phases([PHASE0])
    @spec_test
    @single_phase
    def the_test(spec):
        try:
            yield from case_fn()
        except SkippedTest as e:
            if context.is_pytest:
                pytest.skip(str(e))
            raise

    return (the_test, f"test_{handler_name}_{suite_name}_{case_name}")
```

Seven decorators stacked on a function whose body is a `try/except`
that yields from a closure. `@with_phases([PHASE0])` is **vestigial
for fork-agnostic SSZ tests** — SSZ format primitives don't change
across forks; the decorator is there because the spec_test machinery
requires it. `@only_generator` flags the test as not-runnable under
normal pytest invocation — it only emits during vector generation.
The `try/except SkippedTest` plus the `if context.is_pytest`
conditional hard-codes the dual-mode flag covered in
[decorator-stack.md](decorator-stack.md).

### Frame-inspection-driven test registration (`test_ssz_generic.py:75–77`)

```python
for _handler_name, _suite_name, _cases_fn in test_case_fns:
    for _case_name, _case_fn in _cases_fn():
        _template_ssz_generic_test(_handler_name, _suite_name, _case_name, _case_fn)
```

`template_test` (in `tests/infra/template_test.py:37–55`, covered in
[helper-layer.md](helper-layer.md)'s critique of `tests/infra/`)
uses `inspect.currentframe()` to register each generated function in
the *caller's* module namespace via `setattr`. The 10 imported case
modules each generate dozens to hundreds of cases at module-import
time. Pytest discovers them via the names in the module — no
collection hook, no conftest registration, just frame-injected
attributes.

### `safe_lambda` rejects closures for an undocumented reason
(`ssz_test_case.py:9–15`)

```python
def safe_lambda(fn: Callable):
    code = fn.__code__
    if code.co_freevars:
        raise ValueError(
            f"Multi-threading requires all variables to be captured: {list(code.co_freevars)} in {code.co_filename}:{code.co_firstlineno}"
        )
    return fn
```

`safe_lambda` inspects `__code__.co_freevars` and refuses any
function with closed-over variables. The error message says
"multi-threading requires all variables to be captured", but neither
the framework nor the test driver runs case functions on multiple
threads as far as the file shows. The check is a runtime guard for a
constraint that is real (lambdas with closures over mutable loop
variables in `for x in xs: yield (..., lambda: f(x))` capture by
reference and produce the wrong test) but the *stated reason* is
incidental at best. The cases files (`ssz_basic_vector.py`,
`ssz_uints.py`, etc.) are full of `lambda rng, mode=mode, typ=typ,
length=length: ...` — explicit default-argument captures because
`safe_lambda` rejects implicit closures.

### Vestigial `@with_phases([PHASE0])` (`test_ssz_generic.py:38`)

SSZ format primitives are fork-agnostic. The `@with_phases([PHASE0])`
decorator forces every test through a fork-specific code path
([fork-registration.md](fork-registration.md)) for a test that has
nothing to do with forks. It exists because `@spec_test` expects it.
Tells you something about the inflexibility of the decorator stack.


Each cases file follows the same shape: define type-specific
generation helpers (e.g. `basic_vector_case_fn`), iterate over a
matrix of `types × random_modes × lengths`, yield
`(name, valid_test_case(value_fn))` for each combination. Iteration
patterns are similar but not shared via a single helper — each file
re-implements its own `for typ in TYPES: for length in LENGTHS:
...` loops with type-specific edge cases interleaved.

## Critique / inventory

### The library-as-its-own-oracle problem

The published SSZ generic test suite is the inter-client conformance
spec for SSZ. Other clients (Lighthouse, Lodestar, Prysm, Teku,
Nimbus) consume these vectors and validate that their independent
SSZ implementations agree. That cross-client validation is the *real*
oracle — the consensus-specs test framework only confirms that the
Python implementation is internally consistent. This is acceptable in
practice as long as multiple independent implementations exist and
disagree noisily when the Python lib is wrong.

The dangerous shape is when the test framework's verdict is treated
as authoritative *before* cross-client comparison. The
`ValueError("Invalid X data should not deserialize")` raised at
`ssz_test_case.py:46–48` says, in effect, "the library accepted what
I told it to reject" — which can mean two things: (1) my
`bytez_fn` is buggy, generating valid bytes accidentally; (2) the
library is buggy, accepting truly invalid bytes. The test author
must rule out (2) before publishing. There is no documentation or
checklist for that ruling-out step.

For valid cases, the round-trip property is *necessary* but not
*sufficient*. Necessary: if it fails, definitely a bug. Not
sufficient: passing only proves self-consistency. The byte
representation could be wrong by some property the test never
checks. This is a real concern in spec testing — a wire format is
defined by an *external* document (the SSZ spec), and a serialiser
is correct iff its outputs match that document. Round-tripping is
about implementation invariants, not specification conformance.

The mitigation is differential testing across client implementations.
Three honest acknowledgements: (a) that mitigation is real and works;
(b) it works *post-publication*, when the vectors have already been
shipped; (c) cross-client testing depends on the publishing client
being a *participant*, not the *referee*.

### The seven-decorator chain

Each generated test is wrapped by `@template_test`, then internally
by `@manifest`, `@only_generator`, `@with_phases([PHASE0])`,
`@spec_test`, and `@single_phase` — same surface as the spec-test
decorator stack covered in [decorator-stack.md](decorator-stack.md),
plus three SSZ-specific layers (`@only_generator`, `@manifest`, the
`@template_test` registration). The generated tests yield via the
3-tuple protocol (`("name", "kind", value)`) covered in
[decorator-stack.md](decorator-stack.md) and
[markdown-as-source-of-truth.md](markdown-as-source-of-truth.md);
the kinds here are `"data"`, `"ssz"`, `"meta"` — three of the four
canonical kinds, identifiable by string.

The vestigial `@with_phases([PHASE0])` is the most striking detail:
SSZ is fork-agnostic, but the test infrastructure can't register a
test that doesn't claim a fork.
[fork-registration.md](fork-registration.md) covers why fork
registration is N-place; this is where it bleeds into a domain that
shouldn't need it at all.

### Frame-inspection registration

[helper-layer.md](helper-layer.md) and the §2 finding "Decorator-based
template test registration via frame inspection" cover the
`template_test` machinery in `tests/infra/template_test.py`. SSZ
generic is its largest consumer. The 10 cases modules each iterate
their generators at *module import time*, calling
`_template_ssz_generic_test(...)` for each generated case, which
calls `setattr(caller_module, name, generated_fn)`. Pytest discovers
the tests as if they were ordinary functions in `test_ssz_generic.py`.

The cost: anything that needs to find a test by name has to know
that the test was synthesised at import time. IDE jump-to-definition
lands on `_template_ssz_generic_test`, not on the case generator
that produced the input. Test failures show generated names like
`test_basic_vector_valid_vec_uint64_8_mode_random` whose source is
two layers of indirection away.

### `safe_lambda` is a fragility wart, not a multi-threading
requirement

The check at `ssz_test_case.py:11–13` and the explanation
"Multi-threading requires all variables to be captured" don't quite
match. The actual concern is the late-binding-closure trap in
Python:

```python
for length in [1, 2, 3]:
    yield (f"case_{length}", lambda: case_fn(length))  # all three lambdas see length=3
```

This bug is real and `safe_lambda` does prevent it (by forcing
explicit `lambda length=length: ...` defaults). But the framing as a
multi-threading concern is misleading — the same bug bites a
single-threaded loop. A linter rule (ruff `B023` "function-uses-loop-
variable") catches this at lint time without a runtime guard. Or, in
a typed framework, a `Callable[[Random], View]` signature plus
explicit currying would make the intent clear.

The cases files
(`ssz_basic_vector.py:54–60`, `ssz_uints.py`, etc.) all use the
explicit default-argument idiom (`lambda rng, mode=mode, typ=typ,
length=length: ...`) because of `safe_lambda`. This produces lambdas
with five+ default arguments that exist purely as a workaround for
the runtime check. Inappropriate Intimacy
(Fowler, *Refactoring*, 1999, p. 85) between the framework's
multithreading-safety check and the cases' lambda-construction.

### Ten cases files reimplementing the same iteration pattern

Each cases file's `valid_cases()` and `invalid_cases()` functions
follow the same three-loop pattern (types × random_modes × lengths),
with type-specific tweaks. There is no shared `iter_basic_cases`
helper. Adding a new generic edge case (e.g. "all values truncated
to half their byte length") means edits in 10 files. Shotgun Surgery
(Fowler, *Refactoring*, 1999, p. 79) inside the test-vector
generator.

### Hard-coded `INVALID_BOOL_CASES` cross-imported between cases
files

`ssz_basic_vector.py:21` imports `INVALID_BOOL_CASES` from
`ssz_boolean`. Cross-file constant sharing among cases is undeclared
in the framework — there's no protocol for "this set of edge cases
applies to other types" — it's just import-and-use.

## Named anti-patterns

- **Tautological oracle / library-as-its-own-spec** — the test
  framework's definition of "invalid" is what the library rejects.
  Beck (*Test-Driven Development*, 2002, FIRST) — Self-validating
  tests need an oracle independent of the system under test;
  Feathers (*Working Effectively with Legacy Code*, 2004) calls
  this a "test seam" failure when the seam is the implementation
  itself.
- **Round-trip property as sole correctness check** — `assert
  deserialize(serialize(x)) == x` proves implementation self-
  consistency, not specification conformance. A real concern in
  spec testing where the wire format is defined by an external
  document.
- **Decorator soup** (Fowler, "Comments-as-Deodorant" applied at
  the decorator layer; cross-cuts
  [decorator-stack.md](decorator-stack.md)) — seven decorators on
  every generated test, ordering rules implicit.
- **Stringly-Typed yields** (Fowler, *Refactoring*) — `("kind",
  "data" | "ssz" | "meta", value)` tuples, validated by string
  matching downstream.
- **Vestigial scaffolding** — `@with_phases([PHASE0])` for fork-
  agnostic tests; cross-cuts
  [fork-registration.md](fork-registration.md).
- **Frame-inspection registration** — covered in
  [helper-layer.md](helper-layer.md); SSZ generic is the largest
  consumer.
- **Shotgun Surgery** (Fowler, *Refactoring*, 1999, p. 79) — 10
  cases files reimplementing the same iteration pattern; adding a
  new generic edge case is an N-place edit.
- **Inappropriate Intimacy** (Fowler, p. 85) between `safe_lambda`'s
  closure check and every lambda in the cases files.
- **Catch-everything** (`except Exception`) at
  `ssz_test_case.py:39` — every error type counts as "library
  rejected this", masking distinctions between
  expected-rejection-because-format-violation and
  unexpected-error-because-bug.
- **Self-validating-tests violation** (Beck, FIRST) — the oracle is
  the SUT; only differential testing post-publication catches the
  failure mode where both serialiser and deserialiser are wrong in
  the same direction.

## Comparable contrast — caveat

Neither comparable has a published cross-client SSZ test suite
identical to consensus-specs's, so a direct contrast isn't
available, but both differ in how they would approach the underlying
problem:

- **`leanSpec/src/lean_spec/types/`** — types are Pydantic models.
  Serialisation has explicit invariants checked by Pydantic
  validators (e.g. `Bytes32` is exactly 32 bytes, validated at
  construction). The "validity" of a byte string is decidable
  *without* round-tripping through a deserialiser — the type's
  validator is the oracle, separate from the codec. Tests can
  assert `Bytes32(b"\x00" * 31)` raises `ValidationError` because
  the type *says* it must be 32 bytes; the test doesn't ask the
  codec what it thinks.
- **`execution-specs/packages/testing/src/execution_testing/test_fixtures/`**
  — fixture base classes use Pydantic with `@field_serializer`
  annotations. The serialised form is *defined* by the model's
  serialisation rules, not discovered by the codec. Round-trip
  tests still exist but they test the codec against an
  independently-specified format.

The shape both comparables converge on: **the format is a typed
model**, separate from the codec; the codec is tested against the
model. The published test vectors come from instantiating the
model and emitting its known serialisation, not from generating
random values and trusting the codec to label them.

This pattern requires an authored format-spec (in code), which
[markdown-as-source-of-truth.md](markdown-as-source-of-truth.md)
covers as a foundational issue. SSZ-generic-as-it-stands inherits
the consensus-specs decision to author specs as markdown plus
implement them as Python; the vectors are the only artefact, the
implementation is the only oracle.

## Why this is load-bearing

The published `ssz_generic` vectors are the canonical inter-client
SSZ conformance test suite for Ethereum consensus. Five+ client
implementations consume them. A bug in
`eth_consensus_specs.utils.ssz` that affects vector contents
propagates to:

1. **Other clients are blamed when they disagree.** A vector that
   says "these bytes are invalid" because the Python library
   rejected them — but those bytes are actually valid SSZ — would
   tell every other implementation it's wrong when it accepts them.
2. **Spec drift.** Once a vector is published, fixing the
   underlying library bug requires re-publishing, which clients
   have to re-validate, which exposes the gap between
   "implementation says X" and "spec says X".
3. **Future generic types inherit the framework's tautological
   shape.** Adding `progressive_list` and `compatible_union`
   recently extended the same pattern. Each new SSZ feature gets
   its cases file, its 50–250 lines of generation infrastructure,
   and its self-validation contract.

This is *not* a "Most consequential finding" because the test surface
is bounded (SSZ format primitives, 1 313 lines of code), the tests
work in practice (cross-client testing has caught real issues), and
the library has been extensively used. But the architectural shape
matters: a future maintainer adding a new SSZ generic type, or a
future spec update changing what "invalid" means at the format level,
inherits this pattern and either propagates it or has to fight it.

## What fixing it would entail

A sketch (not a design):

1. **Author hand-computed vectors for invalid cases.** A subset of
   the published "invalid" vectors should come with bytes computed
   by hand or by an independent reference implementation, with a
   docstring explaining *why* those bytes are invalid by the SSZ
   spec (not by the library's behaviour). The framework can
   continue to generate additional cases via the existing
   pattern, but the floor of correctness is anchored in
   independently-verified examples.
2. **Make the invalidity reason explicit.** Replace
   `bytez_fn() -> bytes` with `bytez_fn() -> InvalidCase` where
   `InvalidCase = (bytes, reason: InvalidReason)` and
   `InvalidReason` is an enum (`LENGTH_MISMATCH`,
   `OUT_OF_RANGE_VALUE`, `MALFORMED_BITLIST`, etc.). The test
   asserts the library raises a *specific* exception type matching
   the reason. The published vector includes the reason as
   metadata. Granular oracles replace the catch-everything pattern.
3. **Replace the round-trip property with property-based testing
   keyed on the spec.** Hypothesis-style generators that emit
   values *valid by construction* against a typed model
   ([helper-layer.md](helper-layer.md) recommends this for
   helpers); the codec is then tested for: (a) round-trip,
   (b) equality with hand-computed bytes for canonical cases,
   (c) refusal of mutated invalid bytes. The current framework's
   `valid_test_case` is one of the three.
4. **Drop `safe_lambda` and the closure-rejection runtime check.**
   Replace with a ruff lint rule (`B023`) that catches the same bug
   at lint time. The cases files lose ~10–20% of their lambdas (the
   `mode=mode, typ=typ, length=length` defaults).
5. **Drop `@with_phases([PHASE0])` from SSZ-generic tests.** The
   decorator stack should accommodate fork-agnostic tests
   ([fork-registration.md](fork-registration.md) covers the broader
   issue).
6. **Consolidate the cases iteration pattern.** Ten cases files
   each implement `for typ in TYPES: for length in LENGTHS: for
   mode in MODES: ...` independently. A single `iter_basic_cases`
   helper, parameterised by the type-specific bits, removes the
   Shotgun Surgery footprint.
7. **Replace the 3-tuple yield protocol with a typed result.**
   Cross-references
   [decorator-stack.md](decorator-stack.md) step 4 — the same
   `("name", "kind", value)` tuples flow through this code, with
   the same string-matching downstream.

**Pytest-plugin / fixture angle.** Several of the smells dissolve
into a proper plugin design:

- The seven-decorator chain becomes the
  [decorator-stack.md](decorator-stack.md) plugin's standard
  `@pytest.mark.spec_test` + parametrize fixtures.
  `@only_generator` becomes `@pytest.mark.skip_unless("generator")`
  with a custom collection hook.
- `valid_test_case` and `invalid_test_case` become factory fixtures.
  A test takes `valid_case` and `invalid_case` as parameters; the
  fixture handles the round-trip plus the granular-exception-type
  assertion.
- The 10 cases files become fixture parameters. Hypothesis (or
  pytest-cases) drives the iteration over `(type, mode, length)`
  combinations, replacing the hand-rolled triple-loop in each
  cases file.
- `template_test`'s frame-inspection registration disappears
  entirely — pytest's standard `pytest_generate_tests` hook
  parametrises the test function explicitly, with names visible to
  IDE jump-to-definition.

The pytest-plugin angle is the larger half of the refactor here.
Steps 1, 2, 3 are about correctness (oracle independence). Steps 4,
5, 6, 7 are about replacing ad-hoc machinery with framework
standards.

## A note on `ssz_static/`

The sibling directory
`tests/core/pyspec/eth_consensus_specs/test/phase0/ssz_static/`
follows a similar shape — generated reference vectors for fixed-size
SSZ types (BeaconState, BeaconBlock, etc.). The tautological-oracle
pattern likely applies there too, but in a different form (the
type's structure is fixed, so "invalid" cases are different from
the generic case). A future investigation should treat the two as
likely siblings sharing the same architectural decisions.
